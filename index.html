Зачем меняем

Высокий TTM, расфокус, выгорание. Аналитику никто не ревьюет: где-то избыточна (лишние схемы БД), где-то пропущены пользовательские сценарии.

Высокий TTM

Задача идёт до клиента дольше спринта, релизы расползаются по спринтам.

// FOCUS

Людей дёргают на сторонние задачи внутри спринта. Ритм рваный.

// ANALYSIS

Слабая аналитика

Не ревьюется, есть ненужные схемы БД, пропущены сценарии пользователя.

Меняем три вещи

01 ▸ направления

develop / run

Делим работу: develop — новое, run — текущее, поддержка, рефакторинг. 65% / 35% людей.

02 ▸ фокус

V-TEAM

Внутри develop — команда на одну задачу. Не переключаем на другое внутри 2-недельного спринта.

03 ▸ качество входа

Гейт DoR

Задача без аналитики и проработки не идёт в спринт. Гейт держат CPO + тестлид.

Структура направлений

Два направления

// DEVELOP · 65% · ведёт CPO

Новое и крупное

  • fullstack-команда: 1 BE + 1 FE + 1 QA + 0.5 SA + дизайнер по запросу. Фронт >13 SP.
  • backend-команда: 1 BE + 1 QA + 0.5 SA. Фронт ≤13 SP; FE-лид подключается из run.
// RUN · 35% · ведёт CTO

Текущее и техдолг

  • 1 BE + 1–2 FE + 1 QA + 0.5 SA.
  • Рефакторинг, поддержка, баги, мелкие доработки.
  • Держит текучку в стороне от develop.

// фокус06 / 16
Фокусная команда

V-TEAM — одна задача на спринт

  • Формируется на 1 спринт. Никаких переключений на другие задачи внутри спринта.
  • Живёт до релиза задачи в прод — даже если состав по ротации поменялся, доводит свою задачу.
  • Один человек = одна V-TEAM = одна задача в моменте.
  • Дейлик у V-TEAM свой, отдельно от run.
  • Закончил задачи или ждёшь долгого ответа QA → по согласованию уходишь в run на техдолг.
  • Прерывают только P0 / P1. P2 и ниже — мимо V-TEAM.

// качество входа08 / 16
Гейт DoR

Без проработки — не в спринт

DoR-Hard

// блокирует вход

Проверяют CPO + тестлид на груминге (ср нед 2). Кворум 2 из 2. Нет полного — задача возвращается.

  • H1Цель, гипотеза, бизнес-ценность
  • H2Согласовано с ИБ / смежниками / юристами
  • H3Эпик + декомпозиция задач + оценка
  • H4Решение по смежным системам (даты ИФТ)
  • H5DoD эпика прописан
  • H6DoD спринта прописан
  • H7Каркас спеки + ключевая диаграмма (BPMN/Sequence)

DoR-Soft

// добивает V-TEAM

Контрольная точка — чт / утро пт нед 1. Флаг на дейлике → CPO решает в тот же день.

  • S1Детализация спеки до запросов/ответов — SA
  • S2Тест-кейсы и тест-документация — QA + SA
  • S3Финализирован дизайн — дизайнер + FE
  • S4Финальный копирайтинг — SA + продакт
  • S5Правила продуктовой аналитики — SA
  • S6Что покажем на демо — V-TEAM

Триаж и SA-конвейер

Кто готовит задачи

SA — 2 человека и бутылочное горлышко всего процесса. Триаж и буфер защищают конвейер подготовки.

// ТРИАЖ · раз в квартал · CPO + CTO + SA

Тегируем каждую задачу

  • Направление: develop / run
  • Команда: fullstack / backend
  • Размер анализа: маленькая ≤32 SP · большая >32 SP
// КАПАСИТИ SA · 65 SP / спринт
32 SP · подготовка 18 SP · поддержка V-TEAM 15 SP · встречи + run

Буфер: всегда 3 задачи на разной стадии — в разработке, в финализации, в концепте. Страховка от срыва аналитики.

Схемы подготовки SA

Маленькая — 2 спринта, большая — 3

Схема А · маленькая задача (≤32 SP анализа)
N · подготовка

Кик-оф концепта, спецификация, диаграммы, согласования ИБ. Ср нед 2 — груминг + DoR-гейт (кик-оф V-TEAM совмещён).

N+1 · разработка

Задача в V-TEAM. SA в команде первые 3 дня + по запросу, добивает Soft.

Схема Б · большая задача (>32 SP анализа)
N−1 · концепт

Первичная аналитика, концепт с CTO, старт ИБ, при кросс-команде — арх-ревью. Черновая постановка + уточнённая оценка.

N · финализация

Спецификация. Ср нед 1 — отдельный кик-оф V-TEAM. Финализация Soft, груминг + DoR-гейт.

N+1 · разработка

Задача в V-TEAM. SA первые 3 дня + по запросу.

// исключение12 / 16
Если не успели

Задача не влезла в спринт

Это исключение. План рассчитан так, чтобы укладывались. Бывает из-за ошибки в оценке или новых вводных по ходу.

// что делаем
  • Поднимаем флаг на дейлике сразу, не в последний день.
  • Декомпозируем хвост → S-задача той же V-TEAM на следующий спринт.
  • Релиз сдвигается. Иное решение принимает CPO.
// как минимизируем риск
  • Декомпозируем на груминге, а не «придумаем по ходу».
  • Обсуждаем три уровня скоупа: нужно для клиента / без чего не в прод / nice to have.
  • Не «впихиваем» в спринт всё подряд.
